**Erstellt:** 2026-06-12

## 2. APK direkt teilen (Sideloading) — was wir heute haben

**So funktioniert es:** APK per WhatsApp/E-Mail/Cloud/USB schicken; Empfänger muss
„Installation aus unbekannten Quellen" für die jeweilige App (z. B. den Dateimanager) erlauben.

**Reibung reduzieren statt nur Datei verschicken:**
- Festen **Download-Link + QR-Code** bereitstellen (siehe §3, GitHub Release / Pages).
- Kurze **Einbau-Anleitung** beilegen (3 Schritte: herunterladen → „Trotzdem installieren" →
  ggf. unbekannte Quellen erlauben).

**Wichtig:**
- Neue Versionen installieren sich **über** die alte (kein Datenverlust) — **nur** wenn mit
  **demselben Keystore** signiert. → Keystore niemals verlieren (§7).
- **Kein Auto-Update**: jede neue Version muss erneut verteilt + manuell installiert werden.
- Play Protect kann bei sehr seltenen Apps eine Warnung zeigen (kein Fehler, nur Hinweis).

**Gut für:** Familie, Freunde, kleinen Testkreis — sofort, kostenlos, ohne Konto.

## 3. GitHub Release + Obtainium — Auto-Update ohne Store *(Geheimtipp)*

**Idee:** Jede Version als **GitHub-Release** im bestehenden Repo veröffentlichen (APK als Asset).
Nutzer installieren einmal **Obtainium** (FOSS-App, die APKs direkt aus GitHub-Releases u. a.
aktualisiert), fügen `KoelnThanh/whenopen` hinzu — und bekommen ab dann Update-Hinweise wie aus
einem Store. Kostenlos, kein Google-Konto, passt zur „kein Tracking"-Philosophie.

**Checkliste:**
- [ ] Repo public schalten (für öffentliche Release-Downloads).
- [ ] Release `v1.0.0` anlegen, Tag `v1.0.0`, APK `WhenOpen-v1.0.0.apk` als Asset anhängen.
- [ ] Release-Notes schreiben (Kurzversion der P15-Änderungen).
- [ ] Kurze Obtainium-Anleitung für Nutzer (App-Link + „Quelle hinzufügen").
- [ ] Optional: APK-Dateinamen mit Version pflegen, Tag-Schema `vX.Y.Z` konsequent halten
      (Obtainium erkennt Versionen am Tag/Dateinamen).

**Vorteil:** Auto-Update + öffentlicher, versionierter Download-Link.
**Nachteil:** Nutzer brauchen einmalig Obtainium; für „normale" Laien etwas erklärungsbedürftig.

## 4. F-Droid — der beste Fit für WhenOpens DNA

**Warum stark:** Open Source, ohne Cloud, ohne Google-Dienste, ohne Tracking — genau das
F-Droid-Profil. F-Droid **baut aus dem Quellcode**, liefert Auto-Updates über die F-Droid-App
und ist ein starkes **Vertrauenssignal** in der Privacy-Community.

**Voraussetzungen / zu prüfen:**
- [ ] Lizenz im Repo hinterlegen (z. B. MIT/GPL) — F-Droid braucht eine FOSS-Lizenz.
- [ ] Keine „Anti-Features": keine proprietären Libs/Trackers. (pubspec aktuell sauber:
      go_router, riverpod, home_widget, http, intl, path_provider, url_launcher, share_plus,
      android_alarm_manager_plus, workmanager — alle FOSS; OSM/Nominatim/Overpass FOSS-konform,
      Attribution ist drin.)
- [ ] Reproduzierbarer Build aus dem Repo (F-Droid signiert selbst; eigener Keystore wird für
      diesen Kanal nicht gebraucht).
- [ ] Metadaten-Datei `metadata/com.whenopen.when_open.yml` für `fdroiddata` vorbereiten +
      Merge-Request stellen.
- [ ] Prüfen: WorkManager/AlarmManager-Hintergrundlogik baut ohne GMS (sollte passen).

**Vorteil:** Auto-Updates, hohes Vertrauen, kostenlos, dauerhaft.
**Nachteil:** Einmaliger Einreichungs-/Review-Prozess; Build muss FOSS-rein sein.

## 5. Google Play — der „offizielle" Weg

**Internes Testing (empfohlener Einstieg):**
- Bis **100 Tester** per E-Mail/Google-Konto, **kein** Produktions-Review, Auto-Updates über Play.
- Ideal für Familie/Freunde/kleinen Kreis **ohne** die Produktions-Hürde (§unten).

**⚠️ Stolperstein Produktion (neue persönliche Konten):**
Google verlangt vor der Produktion ein **geschlossenes Testing mit min. 12 Testern über 14 Tage**.
Firmen-/Organisationskonten sind ausgenommen. → **Internes Testing umgeht das, Produktion nicht.**

**Für Produktion zusätzlich nötig:**
- [ ] Play-Developer-Konto (**$25** einmalig) — Konto-Inhaber klären (lt. Notizen: Auftraggeber Marius).
- [ ] **Datenschutz-URL** öffentlich hosten (`docs/privacy-policy.md` → GitHub Pages).
- [ ] **Data-Safety-Formular** ausfüllen (einfach: keine Datenerhebung, lokal).
- [ ] Content-Rating-Fragebogen.
- [ ] Aktuelles **Target-API-Level** (Play-Frist beachten).
- [ ] Store-Listing (`04-Release/play-store-listing.md` ist vorbereitet) + Screenshots.
- [ ] AAB hochladen (für Play **AAB statt APK**: `flutter build appbundle --release`).
- [ ] Play App Signing aktivieren (empfohlen — Upload-Key bei Verlust ersetzbar).

**Vorteil:** maximale Reichweite, Vertrauen, Auto-Updates, Discoverability.
**Nachteil:** Kosten, Auflagen, 12-Tester-Regel für Produktion bei neuem Privatkonto.

## 6. Weitere Kanäle (nachrangig)

- **Amazon Appstore / Samsung Galaxy Store / Aptoide / APKPure** — Nischen, mehr Reibung,
  weniger Vertrauen. Nur bei konkretem Bedarf.
- **Eigene Download-Seite (GitHub Pages)** mit Button + QR — gute Ergänzung zu §3, hostet auch
  gleich die Privacy-URL für Play.
- **In-App-Update-Check** (Feature): App fragt periodisch GitHub-Releases-API nach neuer Version
  und bietet Download an. Vermeidet Obtainium, fügt aber gelegentliche Netzwerk-Calls hinzu
  (opt-in halten, passt nur bedingt zur „rein lokal"-Linie). Eher später, falls überhaupt.

---

## 7. Über allem: Keystore sichern ⚠️

Egal welcher Weg (außer F-Droid, das selbst signiert): **`whenopen-release.jks` sichern.**
Verlust = **keine Updates** mehr möglich (selbe App-ID lässt sich dann nicht mehr aktualisieren).
Ausnahme: Bei **Play App Signing** ist nur der Upload-Key betroffen und ersetzbar.
→ Keystore + Passwort an **zwei** sicheren Orten ablegen (nicht im Git).

---

## 8. Empfehlung

1. **Kurzfristig (heute/morgen machbar):** Repo public + **GitHub Release v1.0.0** mit signierter
   APK → sofort teilbarer Link/QR; **Obtainium** gibt interessierten Nutzern Auto-Updates.
2. **Mittelfristig:** **F-Droid**-Einreichung vorbereiten — bester Fit für die Privacy-Positionierung,
   kostenlos, dauerhaft, Auto-Updates.
3. **Optional/öffentlich:** **Play – Internes Testing** für einen größeren bekannten Kreis;
   Produktion nur, wenn echte öffentliche Reichweite gewünscht ist (dann 12-Tester-Regel einplanen).

## Hintergrund

Die Persona-Mockups (7 Personas) zeichnen eine **reichere Produktvision** als die schlanke
MVP-Spezifikation (`1.1-spezifikation.md`, 2026-06-01). An mehreren Stellen widersprechen sie der
Spec direkt (Eintragslimit, Filter, Notizfeld). Dieses Dokument hält fest, welche Teile der
Persona-Vision **in den MVP gezogen** und welche **auf v2 verschoben** werden.

Diese Entscheidungen **überschreiben** die betroffenen Punkte in `1.1-spezifikation.md` und
`1.2-plan.md`, bis diese nachgezogen sind.

### Bewusst NICHT im MVP (aus Personas, auf v2 vertagt)

- **„Bei Google prüfen"** (Zeiten abgleichen) als Quick-Action.

- **„Telefonisch"-Typ** als eigener Entitätstyp (im MVP: einfach Kategorie „Telefonisch" + leere Adresse, `24h` als durchgehender Block).

## Datenmodell-Änderung (E9) — konkret

**Entität `OpeningDay` (neu):**

| Feld | Typ | Pflicht | Beschreibung |
|---|---|---|---|
| `wochentag` | Enum (mo…so) | ja | Wochentag |
| `geoeffnet` | Boolean | ja | abgeleitet: `zeiten.isNotEmpty` |
| `zeiten` | Array[{von, bis}] | ja | Liste von Zeitblöcken (HH:MM). Leer = geschlossen. |

`pause_von` / `pause_bis` **entfallen** — eine Pause ist die Lücke zwischen zwei Blöcken.

```jsonc
{ "wochentag": "mi", "geoeffnet": true, "zeiten": [
    {"von":"08:00","bis":"11:00"},
    {"von":"14:00","bis":"16:00"},
    {"von":"17:00","bis":"19:00"}
]}
// geschlossen:   { "wochentag": "sa", "geoeffnet": false, "zeiten": [] }
// durchgehend:   { "wochentag": "di", "geoeffnet": true,  "zeiten": [{"von":"08:00","bis":"18:00"}] }
```

**Neue Entität `Kategorie` (E15):**

| Feld | Typ | Pflicht | Beschreibung |
|---|---|---|---|
| `id` | UUID | ja | stabil; Eintrag & Widget verweisen darauf |
| `name` | String | ja | Anzeigename, frei umbenennbar |
| `farbe` | String (Hex) | nein | für Wiedererkennung in Liste/Widget |
| `sortierung` | Int | ja | Reihenfolge in Liste/Widget (per Drag änderbar) |

Eine kleine `kategorien`-Liste lebt in der JSON neben `eintraege`. „Sonstige" ist eine
implizite, nicht löschbare Auffang-Kategorie (kein eigener Datensatz nötig: `kategorie == null`).

**Entität `Location`** — Feld `kategorie` hält jetzt eine **Kategorie-`id`** (statt Freitext),
optional. `null` → Gruppe „Sonstige". Gruppierung/Filter (Liste + Widget) basieren darauf.

**`isOpenNow` (E9):** geöffnet ⇔ aktuelle Uhrzeit liegt in **irgendeinem** Block des heutigen
Tags. „schließt um" = `bis` des aktuellen Blocks. „nächste Öffnung" = `von` des nächsten
Blocks heute, sonst erster Block des nächsten geöffneten Tags.

## Aktualisierung / Zeitabgleich (E16) — konkret

**Grundeinsicht:** Offen/Zu ist eine **Stufenfunktion**. Sie ändert sich *nur* an Block-Grenzen
(`von`/`bis`) und um Mitternacht (neuer Wochentag). Zwischen zwei Grenzen ist nichts neu zu
berechnen → **kein Polling**, sondern gezielt zur nächsten Grenze aufwachen.

**In der App (Vordergrund):**
- Bei `App-Start` und `AppLifecycleState.resumed`: alle Status gegen `DateTime.now()` neu rechnen (billig).
- Während die App offen ist: **ein** Timer auf `naechsteAenderung(now)` (kleinste kommende Grenze über alle Einträge, sonst Mitternacht) → feuert → neu rechnen → neuen Timer setzen. Kein 60-Sekunden-Ticker.

**Im Widget (separater Prozess):**
1. **Ereignisgetrieben (sofort):** nach jedem Speichern/Bearbeiten/Löschen **und** wenn die App in den Hintergrund geht → frische Widget-Daten schreiben + Redraw (`home_widget`).
2. **Grenzgenau:** `AlarmManager`-Alarm auf die nächste Grenze (z.B. 12:00). Callback rechnet neu, zeichnet neu, plant den **nächsten** Alarm. → „bis 12:00" kippt pünktlich auf „geschlossen", ohne Minutenpolling.
3. **Mitternacht:** als Grenze enthalten (neuer Wochentag, neue Zeiten).
4. **Sicherheitsnetz (von Wetter-Apps übernommen):** `WorkManager` periodisch ~15 min — überlebt Neustart, ist Doze-bewusst, fängt verschluckte Alarme ab. Robuster als das wackelige `updatePeriodMillis`. Zusätzlich auf `TIME_CHANGED`/`TIMEZONE_CHANGED` reagieren (Nutzer stellt Uhr/Zeitzone) → neu rechnen + Alarm neu setzen.

**Abgrenzung zu Wetter-Apps:** Wetter pollt bewusst grob (WorkManager 15–60 min, inexakt), weil Serverdaten sich laufend & unvorhersehbar ändern und „ungefähr aktuell" reicht. Bei WhenOpen sind die Wechselzeiten exakt bekannt und die Deadline hart → wir ergänzen das periodische Netz um den **exakten Grenz-Alarm**. Kein Netzwerk, daher günstiger als Wetter-Polling.

**Hinweise:** „Entsperren-Frische" läuft über den App-Lifecycle (App schreibt Widget-Daten beim
In-den-Hintergrund-Gehen) — reine `USER_PRESENT`-Manifest-Receiver sind ab Android 8 gesperrt.
DST/Zeitzonen sind unkritisch, weil Zeiten als lokale HH:MM-Wandzeit gespeichert und gegen lokale
`now()` verglichen werden. Mögliche Pakete: `home_widget` (Bridge/Redraw) + `android_alarm_manager_plus`
(oder ein schlanker nativer `AppWidgetProvider` + `AlarmManager`) für die Weckungen.

## Offene Detailfragen (für die Umsetzung, nicht blockierend)

- Widget bei sehr vielen Einträgen: Vollbild-Scroll vs. „Top 5 offen" + „Alle …"? → Vollbild scrollt (E10), Klärung am Gerät.
- Exakte Alarme auf Android 12+: ggf. `setExactAndAllowWhileIdle` / Berechtigung nötig; Doze kann verzögern → 30-Min-Netz fängt das (E16). Am Gerät verifizieren.
- Kategorie löschen, die als Widget-Filter gebunden ist: Widget fällt auf „Alle Orte" zurück (Verhalten am Gerät bestätigen).
- Drag-Sortierung der Kategorien (E15): nice-to-have; falls P05 zu groß wird, Reihenfolge zunächst alphabetisch, Drag in v2.

## Überblick

**Problem:** Öffnungszeiten fester, regelmäßig besuchter Orte müssen immer wieder neu gesucht werden — obwohl sie sich kaum ändern.

**Ziel:** Android-App mit Widget, die persönliche Öffnungszeiten lokal speichert und auf einen Blick zeigt, was gerade offen ist. Schnell befüllt, täglich nützlich, ohne Suche.

**Scope MVP:**
- Flutter-App (Android, API 26+)
- Widget als dedizierter Screen (offen / geschlossen Tagesansicht)
- Schnelleintrag-Flow (sequenziell Mo–So)
- Manuelle Dateneingabe + optionaler Google Maps Link
- Lokale JSON-Persistenz
- Kein Backend, kein Login, keine Cloud
- Import-Weg: OSM/Nominatim-Lookup mit manueller Nutzerbestätigung (Research abgeschlossen 2026-06-01)

| Komponente | Verantwortlichkeit | Schnittstellen |
|---|---|---|
| **UI-Layer / Screens** | Darstellung, Nutzerinteraktion, Navigation | ruft Business-Logic auf, zeigt Ergebnisse |

| **OpenStatus-Berechnung** | Bestimmt für einen Ort + Zeitpunkt ob geöffnet/geschlossen | nimmt Location-Objekt, gibt Boolean + Zeitstring zurück |
| **Validierungs-Service** | Pflichtfelder, Zeitkonsistenz (von < bis), Mittagspause sinnvoll | gibt Fehlerliste zurück |
| **LocationRepo** | CRUD für Location-Objekte, JSON lesen/schreiben, Backup | Dart-Klasse mit Methoden: getAll, save, delete, export |
| **Import-Modul** | Öffnungszeiten aus externer Quelle übernehmen | gibt Location-Rohdata zurück, App befüllt Form damit |
| **Android Widget** | Tagesansicht offen/geschlossen auf Home-Screen | liest JSON-Datei via `home_widget` Package |

### Schlüsselentscheidungen

| Entscheidung | Gewählt | Alternative | Begründung |
|---|---|---|---|
| Framework | Flutter | React Native, Kotlin | Erstklassiger Widget-Support via `home_widget`, Cross-Platform offen, beste Vibe-Coding-Dokumentation |
| Persistenz | JSON-Datei | SQLite, SharedPreferences | Einfach, lesbar, direkt teilbar als Export-Datei, ausreichend für 10 Einträge |
| Widget-Kommunikation | `home_widget` Package + JSON-Datei | Platform Channels | Package abstrahiert Android-Widget-Komplexität, JSON-Datei als Shared Storage ist bewährtes Pattern |
| Import-Architektur | OSM/Nominatim-Lookup hinter austauschbarem Interface | Google Places API (ToS-Risiko, zu prüfen), Share-Intent (kein Datenwert), direkt eingebaut | Google Places: Caching-Klausel ist ein bekanntes Risiko, muss vor Einsatz geprüft werden (developers.google.com/maps/terms) — daher nicht als Option gewählt. OSM: Caching laut ODbL erlaubt, kein Rechtsrisiko. Interface bleibt austauschbar für v2. |
| State Management | Provider oder Riverpod | Bloc, GetX | Riverpod ist Flutter-Standard für kleine Apps, gut dokumentiert, Vibe-Coding-freundlich |
| Navigation | go_router | Navigator 2.0 direkt | go_router ist Flutter-Standard, deep links vorbereitet für Widget-Tap |

## Logik-Skelett

```
// ══════════════════════════════════════════════════
// KERNLOGIK 1: "Jetzt geöffnet?" Berechnung
// ══════════════════════════════════════════════════

function isOpenNow(location, currentDateTime):          // E9: Block-Logik
    wochentag = currentDateTime.weekday          // mo/di/.../so
    uhrzeit   = currentDateTime.timeOfDay        // HH:MM

    tag = location.oeffnungszeiten.find(wochentag)

    if tag == null OR tag.zeiten.isEmpty:
        return { offen: false, naechsteOeffnung: findNextOpening(location, currentDateTime) }

    // In irgendeinem Block? (Pause = Lücke zwischen zwei Blöcken)
    for block in tag.zeiten:
        if uhrzeit >= block.von AND uhrzeit < block.bis:
            return { offen: true, schliesst_um: block.bis }

    // geschlossen: nächster Block heute (z.B. nach Pause) oder findNextOpening
    naechsterBlockHeute = tag.zeiten.firstWhere(b => b.von > uhrzeit, orElse: null)
    if naechsterBlockHeute != null:
        return { offen: false, naechsteOeffnung: naechsterBlockHeute.von (heute) }
    return { offen: false, naechsteOeffnung: findNextOpening(location, currentDateTime) }

// naechsteAenderung(now): kleinste kommende Block-Grenze über alle Einträge → Scheduling (E16)

function findNextOpening(location, currentDateTime):
    // Suche in den nächsten 7 Tagen den nächsten geöffneten Tag
    for i in 0..6:
        naechsterTag = currentDateTime + i days
        tag = location.oeffnungszeiten.find(naechsterTag.weekday)
        if tag.geoeffnet:
            return { tag: naechsterTag.weekday, von: tag.von }
    return null  // nie geöffnet (Sonderfall, eigentlich durch Validierung verhindert)


// ══════════════════════════════════════════════════
// KERNLOGIK 2: Widget-Daten aufbereiten
// ══════════════════════════════════════════════════

function buildWidgetData(allLocations, currentDateTime):
    geoeffnet  = []
    geschlossen = []

    for location in allLocations:
        status = isOpenNow(location, currentDateTime)
        if status.offen:
            geoeffnet.append({ name: location.name, bis: status.schliesst_um })
        else:
            geschlossen.append({ name: location.name, naechste: status.naechsteOeffnung })

    return { geoeffnet: sortByName(geoeffnet), geschlossen: sortByName(geschlossen) }

// Widget liest JSON-Datei direkt, ruft buildWidgetData auf, rendert Liste


// ══════════════════════════════════════════════════
// KERNLOGIK 3: Schnelleintrag-Flow
// ══════════════════════════════════════════════════

function schnelleintraegFlow():
    location = neuesLeereLocation()
    location.name = input("Name des Orts")  // Pflicht

    vorschlag = null

    for tag in [mo, di, mi, do, fr, sa, so]:
        zeige("Tag: " + tag)
        zeige("Vorschlag: " + vorschlag ?? "leer")

        aktion = nutzerWahl([ÜBERNEHMEN, ANPASSEN, GESCHLOSSEN])

        if aktion == ÜBERNEHMEN AND vorschlag != null:
            location.oeffnungszeiten[tag] = kopiere(vorschlag)

        if aktion == ANPASSEN:
            von = zeitInput("Öffnet um:")
            bis = zeitInput("Schließt um:")
            mittagspause = optional(zeitInput("Pause von:"), zeitInput("Pause bis:"))
            location.oeffnungszeiten[tag] = { geoeffnet: true, von, bis, ...mittagspause }
            vorschlag = location.oeffnungszeiten[tag]  // wird neuer Vorschlag für nächsten Tag

        if aktion == GESCHLOSSEN:
            location.oeffnungszeiten[tag] = { geoeffnet: false }
            // vorschlag bleibt unverändert (letzter gültiger Wert)

    // Validierung
    errors = validiere(location)
    if errors.isNotEmpty:
        zeigeFehler(errors)
        return  // Speichern blockiert

    LocationRepo.save(location)
    navigiere(Hauptliste)


// ══════════════════════════════════════════════════
// KERNLOGIK 4: JSON Lesen / Schreiben
// ══════════════════════════════════════════════════

function loadLocations():
    try:
        json = liesDatei("whenopen_data.json")
        return parseLocations(json)
    catch ParseError:
        backupDatei("whenopen_backup_" + timestamp + ".json")
        zeigeDialog("Daten konnten nicht geladen werden. App startet leer.")
        return []

function saveLocations(locations):
    json = serializeToJson(locations)
    schreibeAtomar("whenopen_data.json", json)  // write-then-rename für Atomarität
```

**Wo liegt die eigentliche Komplexität?**
- `findNextOpening` über Wochengrenzen hinweg (Sonntag → nächster Montag)
- Widget-Update-Zyklus: Android ruft Widget-Provider auf eigenem Schedule auf — App und Widget teilen JSON-Datei, aber Timing ist nicht deterministisch
- Atomares Schreiben der JSON-Datei (kein Datenverlust bei App-Absturz während Schreibvorgang)
- Vorschlagswert-Logik im Schnelleintrag: letzter gültiger Tag, nicht letzter Tag

**Anpassungen an der Architektur nach diesem Schritt:**
- `home_widget` Package kommuniziert über SharedPreferences + AppGroupId, nicht direkt über JSON-Datei — LocationRepo muss Widget-Daten zusätzlich in AppGroupId schreiben nach jedem Save. Das ist eine bekannte Flutter-Widget-Architektur, ändert aber den Datenfluss leicht.

### Datenmodell (Dart-Klassen Übersicht)

| Klasse | Felder | Zustand |

### Fehlerbehandlung

| Fehlerfall | Verhalten | Zuständig |
|---|---|---|
| JSON-Datei nicht gefunden | Leere Liste zurückgeben, neue Datei beim ersten Save anlegen | LocationRepository |
| JSON-Datei korrupt (ParseError) | Backup anlegen, leere Liste zurückgeben, Dialog anzeigen | LocationRepository |
| Schreibfehler (Disk voll o.ä.) | Fehler loggen, Nutzer informieren ("Speichern fehlgeschlagen") | LocationRepository |
| Pflichtfelder fehlen | Speichern blockiert, Fehlermeldung pro Feld | ValidationService |
| Google Maps App nicht installiert | URL im Browser öffnen als Fallback | DetailScreen |
| Widget-Update fehlgeschlagen | Widget zeigt zuletzt bekannten Stand (kein Crash) | Widget-Provider |
